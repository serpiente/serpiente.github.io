The v4 lives in the B-tree and the buffer manager: every random key lands on a different page, almost never in cache. The sequential one dispatches the index cheaply — its CPU spreads across heap, WAL and generating the uuid.

What to read in each profile

UUID v4 — the B-tree branch is huge

Search for ReadBuffer: the backend spends 58.8% of CPU requesting and evicting pages, because every random key needs a page that's almost never in cache. And _bt_compare climbs to 12.5%: many more comparisons to place the insert.

bigint — the index barely shows up

The insert always lands on the last page (the rightmost one), which is already in cache. The B-tree is dispatched in 24.6% and the CPU spreads across the rest: heap_insert (20%), WAL (16%) and generating the org_id uuid (26%).

Note: this is on-CPU time. Waiting on I/O to read pages from disk is off-CPU and doesn't show up here — that cost appeared in the WAL (~3×) and the time (~2×) of the benchmarks.
